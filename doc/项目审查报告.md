### 问题 1：`ServerPlayer::Send()` 永远返回 false — 服务器无法向客户端发送数据

**文件**: `src/server/physics_server.h` 第 74-79 行

```cpp
bool Send(const NetworkMessage& message) override {
    // 通过 NetworkLayer 发送
    return false;  // ← 硬编码返回 false！
}
```

**深度分析**:
- `ServerPlayer` 类声明继承自 `NetworkConnection` 接口，意图封装每个客户端的网络传输通道
- 但 `Send()` 方法体只有注释和 `return false`，没有任何实际发送逻辑
- 这意味着即使 `PhysicsServer::BroadcastSnapshotToAll()` 构建好了 `WorldSnapshotMessage`，所有客户端的 `Send()` 调用都静默失败
- 整个**物理同步架构的核心通信链路在服务器端完全断裂**
- `Receive()` 同样返回 `nullptr`，连接接收通道也不通

**修复建议**:
- 移除 `ServerPlayer::Send/Receive` 的存根实现
- 在 `PhysicsServer` 中直接通过 `NetworkLayer` 路由消息，而不是通过 `ServerPlayer` 对象
- 或者给 `ServerPlayer` 持有独立的 `NetworkLayer` 实例用于发送

### 问题 2：`OnKCPOutput` 回调未实现 — KCP 的输出被完全丢弃

**文件**: `src/common/network_layer.cpp` 第 76-80 行

```cpp
int NetworkLayer::OnKCPOutput(const char* buf, int len) {
    // Cast through the base class static method, which routes to this instance
    (void)buf; (void)len;
    return 0;  // ← 空实现，KCP 产生的所有数据都被丢弃
}
```

**深度分析**:
- `KCPWrapper::OnKCPOutput` (在 `kcp_wrapper.h` 第 376 行) 是一个静态回调，由 C 语言的 KCP 库直接调用
- 它内部调用 `writer_->OnOutput(buf, len)` 来通知上层
- 而 `NetworkLayer` 通过继承 `KCPWriter` 实现 `OnKCPOutput` 回调
- 但 `NetworkLayer::OnKCPOutput` 的实现完全忽略 `buf` 和 `len` 参数，不做任何 `SendUDP` 调用
- 结果：KCP 层计算重传、做分片、做流量控制后产生的所有 UDP 数据包**全部丢失**
- 这解释了为什么整个网络层实际上无法工作

**修复建议**:
```cpp
int NetworkLayer::OnKCPOutput(const char* buf, int len) {
    SendUDP(reinterpret_cast<const uint8_t*>(buf), len);
    return 0;
}
```

### 问题 3：`NetworkLayer::Connect()` 发送了错误类型的消息

**文件**: `src/common/network_layer.cpp` 第 122-132 行

```cpp
bool NetworkLayer::Connect() {
    if (isServer_) return false;

    // 发送一个 CONNECT_REQUEST 立即
    auto msg = std::make_unique<ConnectAckMessage>();  // ← 错误类型！
    ConnectAckMessage* cm = static_cast<ConnectAckMessage*>(msg.get());

    cm->serverTick = 0;
    cm->latency = 0;
    Send(std::move(msg));  // 发送的是 ConnectAck 而不是 ConnectRequest
```

**深度分析**:

- 但实际创建的是 `ConnectAckMessage`（消息类型 ID = 1，属于 `ServerMessageType::CONNECT_ACK`）

- 客户端发的是 ACK，不是请求 — 协议逻辑完全颠倒
- 正确做法应创建 `PlayerInputMessage` 或新增 `ConnectRequestMessage`，或者至少使用 `ClientMessageType::CONNECT_REQUEST` 类型

**修复建议**:

### 问题 5：`WorldSnapshotMessage::Deserialize` 存在边界检查缺陷

**文件**: `src/common/network_protocol.cpp` 第 59-66 行

```cpp
bool WorldSnapshotMessage::Deserialize(const uint8_t*& data) {
    std::memcpy(&snapshotId, data, sizeof(snapshotId)); data += sizeof(snapshotId);
    std::memcpy(&tick, data, sizeof(tick)); data += sizeof(tick);
    uint32_t dataSize = 0;
    std::memcpy(&dataSize, data, sizeof(dataSize)); data += sizeof(dataSize);
    if (data + dataSize > data) {  // ← 永远是 true (data + dataSize 只会更大)
        // safety check  // ← 空注释，无任何操作
    }
    stateData.assign(data, data + dataSize);  // ← 未验证 dataSize 是否越界
    data += dataSize;
    return true;
}
```

**深度分析**:
- `if (data + dataSize > data)` 这个检查**永远不会为 true** — 因为 `dataSize >= 0`，`data + dataSize` 永远不会小于 `data`
- 即使反序列化恶意构造的数据（如 `dataSize = 0xFFFFFFFF`），代码也会尝试分配大量内存或读取越界内存
- 同理 `PlayerInputMessage::Deserialize` 和 `ErrorMessage::Deserialize` 也缺少边界检查

**修复建议**:
```cpp
bool WorldSnapshotMessage::Deserialize(const uint8_t*& data) {
    std::memcpy(&snapshotId, data, sizeof(snapshotId)); data += sizeof(snapshotId);
    std::memcpy(&tick, data, sizeof(tick)); data += sizeof(tick);
    uint32_t dataSize = 0;
    std::memcpy(&dataSize, data, sizeof(dataSize)); data += sizeof(dataSize);
    if (dataSize > 65536) return false;  // 上限检查
    if (dataSize > 10 * 1024 * 1024) return false;  // 10MB 上限
    // 检查剩余数据量（需要外部传入总缓冲区长度）
    stateData.assign(data, data + dataSize);
    data += dataSize;
    return true;
}
```

### 问题 6：`Serializer::Serialize(float)` 浮点字节序转换会覆盖输入值

**文件**: `src/common/serializer.cpp` 第 86-91 行

```cpp
bool Serializer::Serialize(float value) {
    uint32_t intRep;
    std::memcpy(&intRep, &value, sizeof(float));  // intRep 现在有 value 的字节
    intRep = HostToNetwork(intRep);               // intRep 现在是大端序
    std::memcpy(&value, &intRep, sizeof(float));  // ← value 被修改为大端序！
    return SerializeRaw(&value, sizeof(float));    // 写入的是已经改过序的 value
}
```

**深度分析**:
- `std::memcpy(&value, &intRep, sizeof(float))` 把经过字节序交换的 `intRep` 写回 `value`
- 在**小端序机器**（x86/x64）上，这意味着 `value` 本身的字节表示被反转了
- 然后 `SerializeRaw(&value, sizeof(float))` 写入的就是反转后的字节
- 虽然**写入缓冲区的结果碰巧是正确的**（因为 `value` 被改了），但这种写法有严重问题：
  1. 违反 const 语义 — 传入的值被意外修改
  2. 在反序列化端，`Deserialize(float)` 也做同样的字节序转换，结果可能正确但意图不清
  3. 如果在非 x86 大端序机器上，两个转换会相互抵消导致错误

**修复建议**:
```cpp
bool Serializer::Serialize(float value) {
    uint32_t intRep;
    std::memcpy(&intRep, &value, sizeof(float));
    intRep = HostToNetwork(intRep);
    float netValue;
    std::memcpy(&netValue, &intRep, sizeof(float));
    return SerializeRaw(&netValue, sizeof(float));
}
```

### 问题 9：`PredictRotation` 使用 `angularVelocity.Length()` 作为旋转角度

**文件**: `src/common/physics_state.h` 第 300-317 行

```cpp
Quat PhysicsObjectState::PredictRotation(float deltaTime) const {
    float angle = angularVelocity.Length() * deltaTime;  // ← 概念错误
    Vec3 axis = angularVelocity.Normalized();
    // ...
}
```

**深度分析**:
- `angularVelocity` 是**角速度**（单位 rad/s），不是旋转角度
- 公式 `angle = angularVelocity.Length() * deltaTime` 在数学上是正确的（角度 = 角速度 × 时间）
- **但是**，只有当角速度方向恒定且物体不受力矩作用时，这个公式才成立
- 在实际物理引擎中，角速度会随时间变化，应该使用更精确的积分方法
- 不过在这个简化实现中，这个公式本身数学上是对的
- **真正的问题是**: 这个方法**从未被调用** — 客户端的 `Predict()` 方法只调用了 `PredictPosition`

**影响**: 功能不完整，非直接 bug

### 问题 10：`NetworkLayer::FlushRetransmits()` 超时时间硬编码

**文件**: `src/common/network_layer.cpp` 第 322-335 行

```cpp
void NetworkLayer::FlushRetransmits() {
    uint64_t now = NowTickMS();
    for (auto it = sendQueue_.begin(); it != sendQueue_.end();) {
        if (now - it->sentAt >= 200) { // 200ms retransmit interval
            if (it->retries >= 3) {
                packetsDropped_++;
                it = sendQueue_.erase(it);
                continue;
            }
```

**深度分析**:
- 200ms 重传间隔和最多 3 次重试是硬编码的
- 对于物理同步应用，200ms 可能太短（导致不必要的重传）或太长（导致卡顿）
- 重试 3 次后直接丢弃消息，没有任何回调通知上层
- 没有基于 RTT 的动态调整 — 对于高延迟网络（如跨洲连接），3 次重试可能远远不够

**修复建议**: 将重传参数配置化，并基于 RTT 动态计算超时

### 问题 14：信号处理函数调用 `Stop()` 可能死锁

**文件**: `src/server/server_main.cpp` 第 17-23 行

```cpp

    if (g_server && g_server->IsRunning()) {
        g_server->Stop();  // ← 在信号处理函数中调用 Stop()
```

- 在信号处理函数中调用 `join()` 和 `mutex` 操作本身也是未定义行为（POSIX）

### 问题 19：`NetworkLayer::Connect()` 被客户端和服务端共用 — 服务端调用无意义

**深度分析**:
- `NetworkLayer` 同时用于服务端和客户端

- `PhysicsServer::NetworkThread` 中不需要调用 `Connect()`，但 `PhysicsClient::Connect()` 中会调用
- 这个设计本身没问题，但 `Connect()` 内部发送的是 `ConnectAckMessage`（问题 3），所以客户端的连接流程是错误的

### 问题 20：`Deserializer::Deserialize(string)` 没有验证 `length` 上限

**文件**: `src/common/serializer.cpp` 第 204-212 行

```cpp
bool Deserializer::Deserialize(std::string& str) {
    uint32_t length;
    if (!Deserialize(length)) return false;
    if (position_ + length > dataSize_) return false;
    str.assign(reinterpret_cast<const char*>(data_ + position_), length);
    position_ += length;
    return true;
}
```

**深度分析**:
- 虽然检查了 `position_ + length > dataSize_`，但没有设置最大长度上限
- 攻击者可以发送 `length = 0xFFFFFFFF`（虽然会被边界检查拦截）
- 但在某些边界情况下，这可能触发 `std::length_error` 异常
- 建议添加 `if (length > 10 * 1024 * 1024) return false;` 之类的限制

### 问题 23：`DeterministicRandom::Next(min, max)` 存在 modulo bias

**文件**: `src/common/deterministic_random.cpp` 第 49-54 行

```cpp
uint32_t DeterministicRandom::Next(uint32_t min, uint32_t max) {
    uint32_t range = max - min + 1;
    return min + (Next() % range);  // ← modulo bias
}
```

**分析**: 当 `range` 不能整除 `UINT32_MAX + 1` 时，`Next() % range` 会使得较小的值出现概率略高。建议使用拒绝采样方法消除 bias。

### 问题 24：`serializer.cpp` 中 `Serializer` 初始化时缓冲区全为零 — 浪费性能
